# Real-World Catalog & Manifest Landscape — Research + Proposed Design
**Date:** 2026-06-15 · **Status:** synthesis for discussion

Per Spencer's direction: don't anchor the design on the one local example (Alex's `MANIFEST.md`). Survey how industry and research organizations actually solve data discovery/documentation, then design *our* authoritative standard from that — treating Alex's as one input among many.

---

## 1. What the real world converges on

| Source | What it is | What we take from it |
|---|---|---|
| **FAIR principles** (Findable, Accessible, Interoperable, Reusable; 2016) | The lingua franca for research data. | Our north star. Critically, FAIR states **"metadata should be accessible even when the data is not"** — that *is* our discovery-decoupled-from-access principle, validated by an established standard. Findable = rich metadata + indexed + IDs. |
| **Datasheets for Datasets** (Gebru et al.) + **Google Data Cards** | A standardized question set documenting a dataset's motivation, composition, collection, preprocessing, uses, distribution, maintenance. | The *human-readable* content of a good card. A "data card" is the lightweight version — the right altitude for us. |
| **Frictionless Data Package** (`datapackage.json` + Table Schema) | A small JSON descriptor that **sits in the data directory**, listing `resources[]` (path/format/schema) and field-level schema (`name`/`type`/`description`, `primaryKey`, `foreignKeys`). | The *machine-readable* layer. File-based, tooling-agnostic, points at data in place — a near-perfect fit for our constraints. `foreignKeys` ↔ our join/linkage layer. |
| **Data catalogs** — Amundsen (Lyft), DataHub (LinkedIn), OpenMetadata | Hosted discovery platforms: entity-aware + faceted search, usage-based ranking, graph lineage, federated governance. | The *metadata model* (entities, facets, lineage, usage) to borrow even in a file-based v0 — and the v2 escalation target. Heavy infra (the same hosting/auth bottleneck), so not v0. |
| **Data Mesh** (Dehghani) | Decentralized architecture: **domain-oriented ownership**, **data as a product**, **self-serve platform**, **federated computational governance** (global rules all domains adhere to). | The architectural spine — and the resolution to "ours vs. Alex's": the **platform owns the standard + rules + index; domains own their products and conform.** Nobody's one format is the mold. |
| **Open Data Contract/Product Standards** (ODCS/ODPS, Linux Foundation/Bitol) | YAML specs where a data product advertises its schema, quality rules, **SLAs/freshness**, stakeholders. | A data product should **declare and verify freshness/quality**, not rely on a human-typed date. Directly answers the "how current is Alex's manifest, really?" concern. |

## 2. Synthesis — the principles our standard adopts

1. **Two coupled layers per collection** (FAIR + Frictionless + Datasheets): a *machine-readable descriptor* (schema, types, keys, freshness) and a *human-readable card* (what/why/caveats/uses/owner). Most of the machine layer is auto-generated; the human layer is curate-once.
2. **Federated governance, not central authorship** (data mesh): *we* define the schema, the rules, the index, the linkage layer, and the validator. *Owners* (Alex, Emma, Conrad, SYK) maintain their products to that standard. This scales past a one-person team without copying any single person's format.
3. **Freshness is machine-verified, not self-reported** (ODCS + our auto-crawl): the catalog records `last_crawled` and the *actual* source mtime, and the nightly diff flags drift. A human `last_updated` is a claim; the crawl is the check.
4. **Discovery is open; access is separate** (FAIR "metadata accessible even when data isn't"): the manifest is world-readable metadata; access status is recorded and routed.
5. **File-based now, catalog-compatible later** (Amundsen/DataHub model): the descriptors map cleanly onto a real catalog's entity model, so v0 is the substrate for a v2 GUI/search service, not a dead-end.

## 3. Proposed PROTECT manifest design

Each catalogued collection is a **"PROTECT data product"** with two files, plus a federated top-level index.

**(a) Machine-readable descriptor** — `dataset.yaml` (Frictionless-influenced), mostly auto-generated by the crawler:
```yaml
id: patient_sample_isolate_linkage
owner: { name, contact, lab }
root: /usr2/people/protect/Arkin_Lab/protect_data/.../patient_sample_isolate_table_v4
access: { readable_by: [groups], status: open|gated, request_to: ... }
tags: [LINKAGE, CLINICAL, ISOLATE]          # facets, from the flag taxonomy
freshness: { last_crawled, source_mtime, cadence_observed }   # machine-verified
resources:                                   # one per file
  - path: patient_sputum_asma_gold_linkage_table_v_4_0.csv
    format: csv ; size ; rows ; 
    schema: { fields: [ {name, type, description}, ... ], primaryKey, foreignKeys }
lineage: { upstream: [...], downstream: [lakehouse table ...] }
```

**(b) Human-readable card** — `CARD.md` (Datasheet/Data-Card altitude), curate-once: what it is & why, how collected/processed, caveats & known issues, recommended uses, how to access, maintainer.

**(c) Federated top-level index** — the entry point + the agent skill: lists every data product (auto-crawled + owner-maintained), supports the flag-taxonomy facets, links to the linkage/join layer. This is the file-based analogue of Amundsen/DataHub's catalog.

**Governance (the rules the platform enforces):** every product must carry a conforming `dataset.yaml` + `CARD.md`; a validator/linter checks conformance; the crawl supplies machine-verified freshness; non-conforming or stale products surface in the nightly diff report.

## 4. How Alex's manifest fits (correctly, this time)
Alex's `MANIFEST.md` is an **exemplary owner-maintained data product** — we mine it for good patterns (YAML frontmatter, directory map, per-script docs, a changelog) and fold those into the standard. But we **raise the bar**: machine-verified freshness instead of a hand-typed date, explicit schema + keys, and FAIR/Datasheet completeness. Over time we bring his manifest into conformance with our standard — not the other way around.

## 5. v2 escalation path
The file-based metadata model maps onto DataHub/OpenMetadata entities, so when the team wants hosted search + lineage + a GUI (and/or Metabase for queryable data), we ingest the same descriptors rather than redo the work.
