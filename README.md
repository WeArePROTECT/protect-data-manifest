# PROTECT Data Manifest

A point-your-Claude-at-it **catalog of all PROTECT data** — where it lives, what's in it, how it
connects, and how to access it. It describes data **in place** (it copies nothing) and is designed
to be read by a Claude agent first, humans second.

> **Pointing a Claude agent here? Start at [`manifest/INDEX.md`](manifest/INDEX.md)** — the catalog —
> or load the skill in [`skill/`](skill/). Everything else below is orientation.

## How an agent uses this manifest
1. **Open [`manifest/INDEX.md`](manifest/INDEX.md).** It lists every collection with a one-line
   `summary`, `tags`, `keywords`, path, and access status, plus a facet index (`tag → collections`).
2. **Match the request** to a collection via its summary / keywords / tags.
3. **Open that collection** under `manifest/collections/<id>/`:
   - `CARD.md` — what it is, **join recipes**, caveats, and *example questions it answers*.
   - `dataset.yaml` — exact files, columns + types, sizes, and machine-verified freshness.
4. **Follow `related` and the join keys** (e.g. `ASMA_id`, `patient_id`, `sample_id`) to combine
   collections. The canonical join layer is `patient_sample_isolate_linkage`.
5. **Honor `access`.** If a collection is `gated`, tell the user it exists, what's in it, and who
   to ask — never imply they can already read it.

## Two files per collection (the standard)
Each collection is a **data product** described by two files, so machine facts and human knowledge
never overwrite each other:

| File | Owner | Content |
|---|---|---|
| `dataset.yaml` | crawler (auto) | paths, sizes, columns + types, access, machine-verified freshness |
| `CARD.md` | human (curate-once) | meaning, provenance, join keys, caveats, example questions |

The full standard and governance rules are in [`templates/README.md`](templates/README.md). The
design rationale (FAIR, Frictionless Data Package, Datasheets, Data Mesh) is in
[`docs/research/`](docs/research/).

## Discovery is open; access is separate
The manifest holds **metadata only** — nothing sensitive in knowing what exists or what columns a
file has — so it is safe for everyone to read, even for data they can't yet open. Whether you can
read the bytes is a separate permission question the catalog records and routes.

## Kept fresh automatically
The catalog is regenerated by the crawler (`crawler/crawl.py` → descriptors, `crawler/build_index.py`
→ catalog), designed to ride a nightly cron. Freshness is **machine-verified** (`last_crawled` +
real source mtime), not a hand-typed date, so stale or undocumented collections get flagged rather
than silently trusted.

## Repository map
```
protect-data-manifest/
├── README.md                       ← you are here (entry point)
├── manifest/
│   ├── INDEX.md                    ← the catalog (agents start here)
│   └── collections/<id>/           ← dataset.yaml (machine) + CARD.md (human), per collection
├── crawler/                        ← crawl.py, build_index.py, collections.yaml
├── templates/                      ← the data-product standard + governance rules
├── skill/                          ← the dedicated "find PROTECT data" skill
└── docs/                           ← roadmap, decisions, research
```

**Status:** v0, in active build. Coverage is growing; see the Coverage line at the bottom of
`manifest/INDEX.md` for which collections have human cards yet.

**Planned / future enhancements (not yet built):**
- **Discovery sweep** — a periodic scan of `/usr2/people/protect` for sizable data directories *not
  yet registered* in `crawler/collections.yaml`, so brand-new data products get surfaced
  automatically instead of waiting for someone to notice and register them. (Today the crawler only
  watches the roots listed in `collections.yaml`; this would close that one blind spot.)
- Per-table schemas for the lakehouse `.txt` exports; scope `source_mtime` to matched resources for
  glob-based collections (avoids false "stale" flags from sibling activity); a push notification on
  freshness changes; eventually a human GUI / search service (v2). See `HANDOFF.md` §7 for the full list.
